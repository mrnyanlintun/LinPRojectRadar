    About the platform & how it works

    What Lin Opus Gubernatio is and how it is built, plus the methods and framework behind every signal and decision.

    What it does

    Lin Opus Gubernatio implements the Public Capital EVM Intelligence Framework (PCEIF): it turns the records a capital program already produces into an accountable, auditable governance recommendation. The pipeline runs in four steps.

    1. 1. Ingests 26 document types — pay applications, schedules, change orders, RFIs, submittals, inspection and compliance reports — extracting the cost, schedule, and risk figures automatically.
    2. 2. Runs 103 signal modules across 11 categories — earned-value analysis, schedule and cost simulation, document risk, system dynamics, and machine-learning anomaly detection.
    3. 3. Fuses the evidence with Dempster–Shafer combination into a single five-state status — Complete, Green, Yellow, Amber, or Red — that surfaces disagreement between signals rather than averaging it away.
    4. 4. Produces a signal-traced recommended action plan — each action tied to the category and module that triggered it, with a named authority and required documentation — which a named human reviewer records. The platform advises; it never decides.

    A separate Technical Auditor maintains a reference corpus — specifications, codes of practice, and user requirements — and audits submittals and drawings against it, returning traceable findings and recommendations for named human review.

    All project data on this site is synthetic. No real project, agency, employer, contractor, or vendor is referenced.

    Architecture

    103 analytical modules across 11 categories run client-side in the browser; a single serverless endpoint brokers persistence and AI, holding every API key server-side.

    Frontend — GitHub Pages (static)

    Vanilla JavaScript, HTML5, CSS3. No framework. All 103 analytical modules run client-side in the browser.

    Backend — Google Apps Script Web App

    Single /exec endpoint. Serverless, running as the owner. Routes requests, holds API keys in Script Properties, brokers persistence and AI. The browser never holds an API key.

    Endpoints: health, list, listslim, get, create, save, setprojectnumber, archive, restore, chat, analyze, extractsignals, overwritesignal, resetsignals, tts, ingestcorpus, audit

    Persistence — Google Drive

    File-based store. Each project is a numbered folder containing project.json. No SQL database. Archived projects retained and restorable.

    AI Engine — Anthropic Claude API

    All AI calls are server-side via Apps Script. Signal extraction, Lin chat, document identification: claude-sonnet-4-6. Technical auditor, executive brief: claude-opus-4-8. Knowledge retrieval: keyword-matched retrieval from the knowledge library.

    Speech — OpenAI TTS-1

    Text-to-speech via tts-1, onyx voice. Server-side via Apps Script.

    Auth — Google OAuth 2.0

    Stage 1 — single authorized user. Google Sign-In via Apps Script session.

    Research context

    Lin Opus Gubernatio is the demonstration artifact for a Doctor of Engineering praxis by Nyan Lin Tun at The George Washington University, Department of Engineering Management and Systems Engineering. The work follows the Design Science Research (DSR) methodology (Hevner, Peffers): the platform is the designed artifact, evaluated through practitioner validation with public-program controls professionals.

    AI explains and summarizes; governance decisions come from explicit rule-based logic, and every recommended action requires named human approval before it is recorded. The platform is not a validated production system.
